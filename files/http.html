各个协议之间的关系

DNS:解析域名对应的IP地址

HTTP:生成对Web服务器的HTTP请求报文

TCP:将客户端的请求报文和服务器的响应报文进行分割,并且可靠的传送给对方,
三次握手:标记SYN,标记SYN/ACK,标记ACK,四次挥手:FIN,ACK,ACK,FIN

IP:增加目的地的MAC地址,转发给链路层

请求报文

请求方法,请求URI,协议版本,可选的请求首部字段,和内容实体构成

响应报文

协议版本,状态码,状态码原因短语,可选的响应首部字段,和内容实体构成

HTTP方法

GET:获取资源

POST:传输实体主体

PUT:传输文件,,因为没有验证机制,一般不使用

HEAD:获取报文首部

DELETE:删除文件,因为没有验证机制,一般不使用

OPTIONS:查询支持哪些方法

TRACE:追踪路径 (容易引发 XST cross-site-tracing,跨站攻击)

CONNECT:要求用隧道协议连接代理

(XST条件)

  1. 需要目标服务器开启 TRACE 参数

  2. 需要一个用来插入XST代码的地方

  3. 目标站点存在跨域漏洞

HTTP报文

使用 CR+LF 作换行符、是一个多行的字符串文本

报文的结构是：报文首部、空行、报文主体

首部字段一般有四种：通用首部字段、请求首部字段、响应首部字段、实体首部字段

常用的内容编码有四种：gzip（GNU zip）、compress（UNIX系统的标准压缩）、deflate（zlib）、identity（不进行编码）

多部分对象集合：
使用多部分对象集合，需要在首部字段加上 Content-type。
multipart/form-data：在web表单文件上传时使用。multipart/byterangges：状态码206响应报文包含了多个范围的内容时使用。

获取部分内容的范围请求：Range：bytes=5001-10000

内容协商（Content Negotiation）：服务器驱动协商、客户端驱动协商、透明协商

HTTP状态码

2XX 成功

200 OK：请求已正常处理

204 No Content：请求处理成功，但没有资源可返回

206 Partial Content：请求部分资源，响应报文应该由 Content-Range 指定范围的实体内容

3XX 重定向

301 Moved Permanently：永久性重定向，表明需要更换书签

302 Found：临时重定向，书签不需要更换

303 See Other：明确表示客户端应该使用GET方法获取资源

当 301 、 302 、 303 响应状态码返回时,几乎所有的浏览器都会把 POST 改成 GET ,并删除请求报文内的主体,之后请求会自动再次发送。

304 Not Modified：该状态码表示客户端发送附带条件的请求时，不满足条件的状况。

307 Temporary Redirect：临时重定向。会遵照浏览器标准

4XX 客户端错误

400 Bad Request：表示请求的报文中存在语法错误。

401 Unauthorized：返回含有 401 的响应必须包含一个适用于被请求资源的 WWW-Authenticate 首部用以质询( challenge )用户信息。

403 Forbidden：服务器拒绝请求

404 Not Found：无法找到请求的资源

5XX 服务错误

500 Internal Server Error：服务器执行请求时发生错误

503 Service Unavailable：超负载或者停机维护

504 Gateway Timeout：网关超时

Web服务器

代理

接收客户端发送的请求，也接受服务器返回的响应

每次通过代理服务器，会附加一个 Via 首部字段

作用：利用缓存技术减少网络宽带的流量，访问控制，获取访问日志

缓存代理：缓存代理会预先将资源的副本保存在代理服务器

透明代理：不对报文做任何加工

网关

网关是转发其他服务器通信数据的服务器

利用网关，可以将HTTP请求转为其他非HTTP协议通信

隧道

隧道作为相距较远的客户端和服务器之间的中转，保持双方通信连接，并确保客户端和服务器进行安全通信的应用程序。如SSh

确保Web安全的HTTPS

HTTP缺点：

  1. 通信使用明文，可能被窃听
  2. 不验证通信方的身份就能遭遇伪装
  3. 无法证明报文完整性，可能已遭篡改（中间人攻击）

HTTPS == HTTP + 加密 + 认证 + 完整性保护

HTTPS 就是身披 SSL 外壳的 HTTP，并且就具有了上面的功能

HTTPS通信步骤：

  1. 客户端发送 Client Hello 报文开始 SSL 通信。
  2. 服务器可以进行 SSL 通信时，会以 Server Hello 报文作为应答
  3. 之后服务器发送 Certificate 报文
  4. 最后服务器发送 Server Hello Done 报文通知客户端，最初阶段的 SSL 握手协商部分结束。
  5. SSL 第一次握手结束以后，客户端以 Client KeyboardInterrupt Exchange 报文作为回应。
  6. 接着客户端继续发送 Change Cipher Spec 报文
  7. 客户端发送 Finished 报文。
  8. 服务器同样发送 Change Cipher Spec 报文。
  9. 服务器同样发送 Finished 报文
  10. SSL建立完成，从此处开始发送 HTTP 请求。
  11. 服务器开始发送 HTTP 响应
  12. 由客户端发送 close_notify 报文断开链接。

确认访问用户身份的认证

通常核对信息的方式：

  1. 密码
  2. 动态令牌
  3. 数字证书
  4. 生物认证
  5. IC卡

HTTP使用的认证方式：BASIC认证、DIGEST认证、SSL客户端认证、FormBase认证

Session管理和Cookie状态管理步骤

  1. 客户端将ID和密码等登录信息放入报文的实体部分，以POST方法发送到服务器.
  2. 服务器会发放用以识别用户的 Session ID。
  3. 客户端接收到从服务器端发来的 Session ID 后，会将其作为Cookie保存在本地。

什么是密码加盐的方式：给密码增加额外的信息，再利用散列（hash）函数计算出散列值后保存。

基于 HTTP 的功能追加协议

消除HTTP瓶颈的SPDY（speedy）

HTTP 瓶颈：

  1. 一条链接上只能发送一个请求。
  2. 请求/响应的首部未经压缩就发送。
  3. 发送冗长的首部，每次都要发送整个页面。
  4. 非强制压缩发送。
  5. 请求只能从客户端开始。

Ajax（Asynchronous JavaScript and XML，异步 JavaScript 与 XML 技术）：是一种有效利用 JavaScript 和 DOM（Document Object Model，文档对象模型）的操作，达到局部 Web 页面替换加载的异步通信手段。

Ajax的核心技术是名为 XMLHttpRequest的API，解决了每次发送整个页面的问题。

Comet解决了客户端发送大量请求来获取信息的问题：当服务器出现更新时，自动向客户端进行推送（Server Push）。

SPDY 以会话层的形式加入，控制对数据的流动，但还是采用HTTP建立通信链接。

SPDY提供的功能：

  1. 多路复用流，通过单一的TCP连接，可以无限制处理多个HTTP请求。
  2. 赋予请求优先级。
  3. 压缩HTTP首部。
  4. 推送功能。
  5. 服务器提示功能。

使用浏览器进行全双工通信的 WebSocket

一旦Web服务器与客户端之间建立起WebSocket协议的通信连接，之后所有的通信都依靠这个专用协议进行。并且服务器还有客户端都可以向对方发送报文。

功能： 减少通信量。WebSocket会希望一直保持连接状态。所以不用像HTTP那样每次都连接。
